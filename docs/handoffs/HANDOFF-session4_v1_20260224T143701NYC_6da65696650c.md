## Architecture Decisions

### Content-Addressable IDs
- `id = sha3(JSON.stringify({...record, parentId, ts}))[0:12]`
- `ts` = precise timestamp in microseconds (prevents collisions)
- `parentId` chain forms Merkle list (tamper-evident)
- Single `_createEntry<T>()` touch point for all session appends

### Tool Execution Categories
1. **Pure** - read, grep, find, ls, pwd - no side effects, parallel OK
2. **Tracked mutation** - edit, write, mkdir, touch, mv, cp - squiggle-gated
3. **Danger zone** - rm, chmod, chown - interactive confirmation (`allow`)
4. **Critical** - .git, ~, / - double confirm (`allow` + `yes`)
5. **Untrackable** - bash - strictly sequential, nukes all read context

### Shadowing Rules
- **Writes invalidate reads** (not vice versa)
- **Bash nukes ALL reads** (can't track what it touches)
- **2-turn grace period** after shadow, then compact
- **Preserve brackets, invalidate body** - keep structure, tombstone content
- **Body UUID** - content hash survives compaction as witness

### Squiggle = Proof of Cognition
- Write/edit requires unshadowed read + squiggle block between read and write
- Model must demonstrate visible reasoning before mutation
- Prevents blind/stale edits

### Batching
- Tool calls should list multiple paths in one call
- Danger zone: one `allow` beats five
- Pure tools: one call enables internal parallelism
- Batch returns individual handles: `§h1, §h2, §h3 = read([a, b, c])`
